# SRS2 — Модуль Deskew для сканированных документов

Версия документа: 0.2  
Статус: draft для production-ready MVP  
Источник: переработка `SRS.md` репозитория `AlexanderKuzikov/DocuDeskew`

---

## 1. Назначение

Модуль `Deskew` предназначен для автоматического устранения углового перекоса одного сканированного документа на светлом фоне.

Модуль принимает grayscale-изображение страницы, определяет угол перекоса документа в диапазоне примерно от `-45°` до `+45°`, выравнивает изображение так, чтобы границы документа и строки текста стали ближе к горизонтали/вертикали, и возвращает выровненный PNG-буфер.

Модуль **не решает** задачи следующего этапа:

- определение читаемой ориентации: верх/низ документа;
- доворот на `90°`, `180°`, `270°`;
- исправление перспективного искажения;
- OCR;
- классификация типа документа;
- извлечение текста.

---

## 2. Область действия

### 2.1. В scope

- Один документ на одном изображении.
- Светлый или белый фон.
- Угловой перекос без сильной перспективы.
- Форматы входа: PNG/JPEG.
- Нормализация входа в 8-bit grayscale.
- Возврат угла, confidence, статуса и выровненного изображения.
- Обработка типовых ошибок входа.

### 2.2. Out of scope

Модуль не обязан корректно обрабатывать:

- несколько документов на одном листе;
- документ, касающийся края изображения;
- сильную перспективу/трапецию;
- поворот на `90°`, `180°`, `270°`;
- фотографии с камеры вместо скана;
- сильный шум, тени, заломы, пятна;
- низкоконтрастные сканы без различимой границы документа;
- документы на тёмном/пёстром фоне;
- восстановление обрезанных краёв документа.

Такие случаи должны завершаться статусом `failed` или `low_confidence`, а не выдавать ложный успешный результат.

## 3. Термины

| Термин | Значение |
|---|---|
| Deskew | Устранение углового перекоса изображения |
| Skew angle | Угол перекоса документа относительно горизонтали |
| Confidence | Оценка уверенности результата от `0` до `1` |
| minAreaRect | Минимальный ограничивающий прямоугольник для набора точек |
| Morphology | Морфологические операции: dilation/erosion |
| Trim | Удаление фона, добавленного поворотом изображения |

---

## 4. Входные данные

### 4.1. Основной вход

```ts
imageBuffer: Buffer
```

Поддерживаемые форматы:

- `image/png`
- `image/jpeg`

Ожидаемый сценарий:

- 8-bit grayscale;
- светлый или белый фон;
- один документ;
- разрешение порядка 150–300 DPI;
- перекос в диапазоне примерно от `-45°` до `+45°`.

Цветные изображения и изображения с alpha-каналом допускаются как предохранительный сценарий: модуль должен конвертировать их в 8-bit grayscale через `sharp.grayscale()`.

### 4.2. Опции

```ts
type DeskewOptions = {
  /**
   * Порог бинаризации градиента.
   * Рекомендуемый диапазон: 0–255.
   * Значение по умолчанию должно быть DPI-aware или документировано для 300 DPI.
   */
  edgeThreshold?: number;

  /**
   * Количество итераций dilation.
   * Должно быть неотрицательным целым числом.
   */
  dilateIterations?: number;

  /**
   * Количество итераций erosion.
   * Должно быть неотрицательным целым числом.
   */
  erodeIterations?: number;

  /**
   * Отступ после финальной обрезки.
   * Рекомендуется задавать в пикселях, но документ должен явно описывать
   * DPI-зависимость или физический размер отступа.
   */
  padding?: number;

  /**
   * Минимальная уверенность для статуса "ok".
   * Значение по умолчанию: 0.75.
   */
  minConfidence?: number;

  /**
   * Максимальный размер входного изображения в пикселях.
   * Защищает от DoS через огромные файлы.
   * Рекомендуемое значение по умолчанию: 50_000_000 пикселей.
   */
  maxPixels?: number;
};
```

## 5. Выходные данные

Результат всегда должен быть строго типизирован:

```ts
type DeskewResult =
  | {
      status: 'ok';
      angle: number;
      confidence: number;
      orientation: 'portrait' | 'landscape';
      deskewedImage: Buffer;
    }
  | {
      status: 'low_confidence';
      angle: number;
      confidence: number;
      orientation?: 'portrait' | 'landscape';
      deskewedImage: null;
      reason: string;
    }
  | {
      status: 'no_document';
      angle: 0;
      confidence: number;
      orientation: null;
      deskewedImage: null;
      reason: string;
    }
  | {
      status: 'unsupported_case';
      angle: 0;
      confidence: 0;
      orientation: null;
      deskewedImage: null;
      reason: string;
    };
```

### 5.1. Семантика статусов

| Статус | Значение | Что делать downstream |
|---|---|---|
| `ok` | Результат можно использовать | Передавать изображение дальше |
| `low_confidence` | Угол найден, но доверять ему рискованно | Логировать, отправлять на ручную проверку или fallback |
| `no_document` | Документ не обнаружен | Не применять deskew |
| `unsupported_case` | Сценарий вне поддерживаемого scope | Не применять deskew |

---

## 6. Ошибки

Модуль должен выбрасывать ошибку только для технических/входных проблем, которые не являются результатом анализа изображения:

```ts
type DeskewError = {
  code:
    | 'INVALID_BUFFER'
    | 'INVALID_IMAGE'
    | 'IMAGE_TOO_LARGE'
    | 'INVALID_OPTIONS'
    | 'PROCESSING_ERROR';
  message: string;
};
```

Примеры:

- пустой буфер;
- не PNG/JPEG;
- повреждённый файл;
- изображение больше `maxPixels`;
- некорректные опции;
- внутренняя ошибка обработки.

## 7. Алгоритм обработки

### 7.1. Нормализация входа

1. Проверить, что `imageBuffer` — непустой `Buffer`.
2. Прочитать изображение через `sharp`.
3. Проверить размер изображения относительно `maxPixels`.
4. Конвертировать изображение в 8-bit grayscale.
5. При необходимости масштабировать изображение вниз для оценки угла, но сохранять исходное изображение для финального поворота.

### 7.2. Предварительная оценка качества

До основного алгоритма необходимо оценить:

- есть ли достаточно контрастных пикселей;
- есть ли признаки документа/границы;
- нет ли чрезмерного шума;
- не является ли изображение почти пустым или однотонным.

Если признаков недостаточно — вернуть `no_document` или `unsupported_case`.

### 7.3. Выделение краёв

1. Применить Sobel 3×3 для горизонтальной и вертикальной составляющих градиента.
2. Вычислить магнитуду:
   - точный вариант: `sqrt(Gx² + Gy²)`;
   - ускоренный вариант: `|Gx| + |Gy|`.
3. Бинаризовать магнитуду порогом `edgeThreshold`.

Результат: бинарное изображение, где белые пиксели соответствуют резким перепадам яркости.

### 7.4. Морфологическая обработка

1. Применить dilation с квадратным структурным элементом 3×3.
2. Применить erosion для восстановления толщины линий.
3. Количество итераций должно зависеть от разрешения или явно документироваться.

Цель — соединить разрывы границы документа, но не превратить текст в доминирующую структуру.

### 7.5. Фильтрация точек

Из бинарного изображения извлекаются координаты белых пикселей.

Допускается:

- прореживание точек при большом количестве;
- удаление мелких компонент шума;
- приоритет длинных непрерывных контуров над мелкими текстовыми фрагментами.

### 7.6. Поиск угла

Вариант MVP:

1. Построить выпуклую оболочку точек.
2. Грубый проход: углы от `0°` до `180°` с шагом `5°`.
3. Точный проход: диапазон найденного угла ±`5°` с шагом `0.1°`.
4. Найти угол минимальной площади ограничивающего прямоугольника.
5. Нормализовать угол в диапазон `[-45°, +45°]`.

Рекомендуемое улучшение для production:

- дополнительно оценивать угол по текстовым линиям через Hough/projection profile;
- сравнивать угол границы документа и угол текста;
- снижать `confidence`, если оценки расходятся.

### 7.7. Расчёт confidence

`confidence` должен рассчитываться явно. Минимальные факторы:

- доля найденных точек, принадлежащих предполагаемой границе;
- длина непрерывных контуров;
- разница между лучшим и вторым лучшим углом;
- симметрия/прямоугольность найденного контура;
- согласованность угла границы и угла текста;
- отсутствие чрезмерного шума.

Рекомендуемые пороги:

| Confidence | Статус |
|---:|---|
| `>= 0.75` | `ok` |
| `0.45–0.75` | `low_confidence` |
| `< 0.45` | `no_document` или `unsupported_case` |

Конкретные пороги должны быть подтверждены тестовым датасетом.

### 7.8. Выпрямление

1. Повернуть исходное grayscale-изображение на найденный угол через `sharp.rotate(angle, { background: 255 })`.
2. Обрезать фон, добавленный поворотом, только если это не удаляет реальные поля документа.
3. Добавить `padding`.
4. Вернуть PNG-буфер.

Важно: `trim()` не должен без контроля удалять законные белые поля документа.

### 7.9. Определение ориентации

Ориентация определяется по ширине и высоте финального ограничивающего прямоугольника или финального изображения:

```ts
orientation = width > height ? 'landscape' : 'portrait';
```

Для квадратного случая выбирается `portrait`, если иное не указано в опциях.

## 8. Требования к реализации

### 8.1. Среда

- Node.js `20+`.
- TypeScript preferred.
- `sharp` — единственная разрешённая runtime-зависимость.
- Без дополнительных нативных аддонов.
- Формулировка «без нативных аддонов, кроме sharp» должна учитывать, что `sharp` сам использует native bindings.

### 8.2. API

```ts
import { deskew, type DeskewOptions, type DeskewResult } from 'docu-deskew';

const result = await deskew(imageBuffer, options);
```

Поддержка:

- CommonJS;
- ESM;
- TypeScript types.

### 8.3. Производительность

Целевой сценарий:

- A4, 300 DPI;
- один документ;
- светлый фон;
- перекос в пределах `[-45°, +45°]`;
- типовая server CPU, указанная в benchmark.

Цель:

- `P50 <= 500 ms`;
- `P95 <= 1500 ms`;
- память: желательно `< 1 GB` на изображение.

Если цель не достигается без native-зависимостей, это должно быть явно зафиксировано как trade-off.

### 8.4. Безопасность

- Ограничение `maxPixels`.
- Ограничение максимального размера буфера.
- Завершение обработки при повреждённом изображении.
- Отсутствие сетевых запросов.
- Отсутствие чтения/записи файлов внутри библиотеки.

### 8.5. Наблюдаемость

Для production желательно добавить опциональный logger/metrics:

- время обработки;
- размер изображения;
- найденный угол;
- confidence;
- статус результата;
- причину отказа.

## 9. Тестирование

### 9.1. Unit-тесты

Обязательные unit-тесты:

- валидация входа;
- валидация опций;
- grayscale-конвертация;
- Sobel-бинаризация;
- dilation/erosion;
- фильтрация шума;
- нормализация угла;
- определение orientation;
- error codes.

### 9.2. Интеграционные тесты

Набор эталонных изображений должен включать:

| Класс | Примеры |
|---|---|
| Clean scan | белый фон, чёткая граница, перекос `-30°…+30°` |
| Noisy scan | JPEG-артефакты, лёгкий шум |
| Text-heavy | таблицы, плотный текст, подписи |
| Margins | документы с широкими белыми полями |
| Low contrast | слабая граница документа |
| Shadow | лёгкая тень по краю |
| Negative cases | пустой лист, однотонный фон, несколько документов |

### 9.3. Метрики качества

Для каждого эталонного изображения должны храниться:

- ground truth angle;
- ожидаемый статус;
- минимально допустимый confidence;
- допустимая погрешность угла.

### 9.4. Критерии приёмки MVP

На тестовом наборе из минимум 100 корректных сканов:

| Метрика | Требование |
|---|---:|
| Успешные `ok` на корректных сканах | `>= 95%` |
| Погрешность угла | `P95 <= 0.5°` для синтетических/идеальных сканов |
| Погрешность угла | `P95 <= 2.0°` для реальных сканов |
| Ложные `ok` на negative cases | `0` |
| Время обработки | `P50 <= 500 ms`, `P95 <= 1500 ms` |
| Нет обрезки документа | визуально/метрически подтверждено на тестовом наборе |

## 11. Контракт с downstream

Downstream не должен интерпретировать `angle: 0` как успешный результат.

Корректная логика:

```ts
const result = await deskew(buffer);

if (result.status === 'ok') {
  use(result.deskewedImage);
} else {
  handleFailedDeskew(result);
}
```

## 12. Риски

| Риск | Вероятность | Влияние | Митигация |
|---|---:|---:|---|
| Текст доминирует над границей документа | высокая | неправильный угол | confidence, фильтрация контуров, сравнение с Hough/projection |
| `trim()` удаляет реальные поля | средняя | потеря части документа | аккуратная логика trim или отказ от агрессивной обрезки |
| Производительность не укладывается в 500 мс | средняя | slowdown pipeline | downscale, benchmark, опциональный native/WASM |
| Negative case возвращает `ok` | высокая без confidence | ложная обработка | status + confidence + negative tests |
| Параметры не работают на 150/300 DPI | средняя | нестабильное качество | DPI-aware defaults |

## 14. Итоговое определение готовности

Модуль считается готовым к production MVP, когда:

- есть `status` и `confidence`;
- negative cases не возвращают `ok`;
- есть тестовый датасет с ground truth;
- есть benchmark;
- есть CI;
- есть package metadata;
- README описывает supported/unsupported cases;
- performance target подтверждён на реальной машине.
